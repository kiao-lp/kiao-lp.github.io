Du Prompt Engineering au Context Engineering

De l’art de poser des questions à l’art de construire des environnements de travail pour les systèmes agentiques.

Positionnement

Le terme Context Engineering a été popularisé par Andrej Karpathy (ex-Tesla, OpenAI) pour désigner le travail de construction du contexte fourni à un LLM — par opposition à la simple rédaction de prompts. Depuis, la communauté autour des systèmes agentiques (Auto-GPT, Claude Code, Cursor, etc.) a largement adopté et enrichi cette idée.

Cet article synthétise ces concepts émergents et les prolonge d’une perspective d’architecte solution : comment structurer l’information pour piloter des systèmes agentiques sur des tâches complexes — avec la même rigueur qu’on appliquerait à une architecture logicielle.

1. Constat : les limites du prompt engineering

Le prompt engineering — l’art de formuler des instructions précises pour obtenir des réponses pertinentes d’un LLM — a été le paradigme dominant de l’interaction homme-IA. Mais l’émergence des systèmes agentiques (Claude Code, Antigravity, Cursor, etc.) révèle ses limites.

Un prompt, aussi bien formulé soit-il, reste une instruction ponctuelle. Or les tâches réelles d’un architecte — concevoir un module, auditer une intégration, documenter un flux — nécessitent un environnement informationnel riche et structuré.

Prompt Engineering Context Engineering
Unité de travail L’instruction L’environnement
Nature Textuelle, ponctuelle Structurée, multi-documents
Portée Une question → une réponse Un projet → des livrables cohérents
Expertise requise Formulation de requêtes Architecture de l’information
Analogie architecture Un ticket Jira Un dossier d’architecture

2. Le Context Engineering : définition

Le Context Engineering est la discipline qui consiste à concevoir, assembler et raffiner un ensemble structuré de documents destinés à servir d’environnement de travail à un système agentique.

Un contexte bien construit ne se limite pas à décrire ce qu’on veut faire. Il inclut :

  • 🎯 Les actions attendues — ce que le système doit produire
  • 🏗️ L’architecture — les contraintes techniques, les patterns, les conventions
  • 📋 Les exigences métier — le domaine, les règles, les cas d’usage
  • 📚 Les documents de référence — spécifications, exemples, glossaires

[!IMPORTANT] Le contexte n’est pas un simple “prompt amélioré”. C’est un artéfact d’ingénierie à part entière, conçu avec la même rigueur qu’une architecture logicielle. On retrouve ici une parenté directe avec les ADR (Architecture Decision Records) et les vues C4 : des documents structurés qui cadrent le travail de ceux qui les consomment.

3. La chaîne de production de contexte

Un contexte optimal ne se construit pas en une seule étape. Il émerge d’une chaîne de production — une succession de maillons où chaque maillon produit un nouveau contexte qui remplace entièrement le précédent.

Le principe fondamental : remplacer, pas accumuler

La mécanique est la suivante : à chaque maillon, on oublie le contexte d’entrée pour ne conserver que le contexte de sortie. Cette substitution est l’idée centrale de la chaîne.

Pourquoi ? Parce que la fenêtre de contexte d’un LLM est limitée (128K–200K tokens selon les modèles, soit environ 75 à 150 pages de texte). En remplaçant le contexte au lieu de l’accumuler, on dispose à chaque maillon de la totalité de la fenêtre pour travailler. Le contexte ne grossit pas : il se transforme.

[!IMPORTANT] À chaque étape, on se place dans le nouveau contexte. Le contexte précédent n’est plus nécessaire — toute l’information utile a été transférée, transformée, dans le nouveau.

Anatomie d’un maillon

Chaque maillon suit un cycle interne en trois phases :

  1. Analyser — comprendre le contexte d’entrée, identifier ce qui est pertinent
  2. Modifier — transformer, restructurer, enrichir ou réduire le contenu
  3. Vérifier — s’assurer que le contexte de sortie est cohérent et complet

Types de maillons

Le but d’un maillon varie selon l’étape de la chaîne. Voici des exemples courants :

Type de maillon But Exemple
Réduire / Nettoyer Éliminer le bruit, les doublons, l’inutile Partir de 13 fichiers de notes brutes → un seul document concentré
Structurer Organiser l’information, créer un plan Transformer un ensemble de concepts en une hiérarchie thématique
Enrichir Ajouter du contenu, développer, illustrer Compléter un plan avec des définitions, exemples et diagrammes
Reformuler Adapter le ton, le niveau, le public cible Réécrire un document technique en guide pédagogique
Spécialiser Extraire un sous-ensemble pour un usage précis Dériver un document d’architecture en spécifications d’API

4. Les mécanismes à l’œuvre

La chaîne de production de contexte tire sa puissance de trois propriétés :

Fenêtre de contexte maximisée

C’est l’avantage stratégique principal. En oubliant le contexte précédent à chaque maillon, on libère la totalité de la fenêtre de contexte pour la tâche en cours. Au lieu de traîner un historique qui s’accumule et finit par saturer la fenêtre, chaque maillon travaille dans un espace neuf avec un contexte sur mesure pour sa responsabilité.

Raffinement progressif

Comme un compilateur transforme du code source en code machine via des passes successives, chaque maillon réduit le bruit et augmente la pertinence du contexte. L’information est progressivement façonnée vers sa forme optimale.

Capitalisation

Les contextes intermédiaires sont des artéfacts réutilisables. Un contexte architectural validé peut servir de point de départ à de multiples chaînes différentes.

5. Limites et trade-offs

Comme toute approche d’ingénierie, le context engineering a ses limites. Un architecte doit les connaître pour choisir le bon outil au bon moment.

Limite Détail
Coût de préparation Construire une chaîne de contexte prend du temps. Pour une question ponctuelle, un prompt direct reste plus efficace. L’investissement se justifie quand la tâche est récurrente ou complexe.
Risque de sur-ingénierie On peut passer plus de temps à structurer le contexte qu’à résoudre le problème. La règle : si le contexte dépasse en effort le livrable attendu, simplifier.
Perte d’information inter-maillons Le remplacement du contexte à chaque maillon repose sur l’hypothèse que le maillon a correctement extrait l’essentiel. Une mauvaise exécution d’un maillon propage l’erreur. D’où l’importance de la validation humaine intermédiaire.
Dépendance à la qualité des sources Un contexte structuré à partir de sources incorrectes produit des livrables incorrects — avec une confiance trompeuse liée à la qualité de forme.
Pas un substitut au jugement Le context engineering améliore la production, pas la décision. L’architecte reste responsable des choix structurants.

[!TIP] Règle empirique : utiliser le context engineering quand le projet nécessite plus de 3 interactions avec un LLM pour produire un livrable cohérent. En dessous, un prompt bien formulé suffit.

6. Outillage pratique

Pour les praticiens, voici les formats et outils qui supportent concrètement cette approche :

Outil / Format Usage
Fichiers CLAUDE.md Instructions persistantes pour Claude Code — un contexte architectural réutilisable entre sessions
.cursorrules Équivalent pour Cursor — conventions de code, patterns, contraintes
Markdown structuré Format naturel pour les contextes : titres, listes, tableaux, blocs de code. Universel et versionnable.
Dossiers de contexte Organiser les fichiers de contexte dans un répertoire dédié du projet (/context, /specs, /docs)
ADR (Architecture Decision Records) Contexte décisionnel — chaque ADR alimente naturellement le contexte d’un agent
Diagrammes C4 en texte Les diagrammes Mermaid/PlantUML intégrés au contexte donnent une compréhension spatiale au LLM

7. Le changement de paradigme

Avant : Prompt Engineering

Après : Context Engineering

Le rôle de l’humain évolue : il n’est plus rédacteur de prompts, il devient architecte de contextes. La valeur se déplace de la formulation de l’instruction vers la conception de l’environnement informationnel dans lequel le système agentique opère.
